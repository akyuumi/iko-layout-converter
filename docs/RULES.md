# **プロジェクト推進ルール**

## **目的**

このドキュメントは、Codex を用いた自律的・半自律的な開発を安定運用するためのルールを定義する。

目的は以下。

- 仕様解釈のブレを減らす
- 実装方針の一貫性を保つ
- 設計判断を蓄積する
- スコープ暴走を防ぐ
- リポジトリの可読性を維持する
- 将来的な保守性を高める

---

# **ドキュメント構成**

プロジェクト管理用ドキュメントは以下に配置する。

```
/docs
```

必須ファイル:

```
/docs
  SPEC.md
  TASKS.md
  DECISIONS.md
  CHANGELOG.md
```

必要に応じて追加:

```
/docs
  BUGS.md
  ROADMAP.md
  IDEAS.md
```

---

# **各ファイルの役割**

## **SPEC.md**

システム仕様書。

役割:

- 要件定義
- システム仕様管理
- 実装対象の定義

記載対象:

- 機能要件
- 非機能要件
- 制約事項
- アーキテクチャ概要
- 画面仕様
- API仕様
- DB設計
- セキュリティ方針
- デプロイ前提

原則:

```
仕様の正は SPEC.md とする
```

実装と SPEC.md が矛盾した場合、原則として SPEC.md を優先する。

---

## **TASKS.md**

実装タスク管理ファイル。

役割:

- 進捗管理
- タスク分解
- 作業順序管理
- 開発状況の可視化

ルール:

- Markdown のチェックボックス形式を使用する
- 完了時は `[x]` に変更する
- 実装中に発見した追加タスクは `Backlog` に追加する
- タスクは実装単位で分割する
- 完了条件を明記する

良い例:

```markdown
- [ ] Googleログインを実装する
  - 完了条件:
    - ログインできる
    - ログアウトできる
    - 未認証時は保護ページへアクセスできない
```

悪い例:

```markdown
- [ ] 認証機能を作る
```

推奨粒度:

```
30分〜2時間程度で完了できるサイズ
```

巨大タスクは禁止。

---

## **DECISIONS.md**

設計判断記録ファイル。

役割:

- なぜその設計にしたかを残す
- 将来の再議論コストを下げる
- トレードオフを明示する
- 制約を共有する

各記録に含める内容:

- 日付
- 決定事項
- 理由
- 影響
- 制約

例:

```markdown
## 2026-05-17: 独自JWT認証ではなく Supabase Auth を採用

理由:

- MVP開発速度を優先
- RLSとの統合が容易
- 運用コスト削減

影響:

- ベンダー依存が増える
- 認証仕様の自由度は下がる

制約:

- 将来的な移行可能性は維持する
```

---

## **CHANGELOG.md**

実装履歴管理ファイル。

役割:

- 人間が読める変更履歴を残す
- Git履歴を補完する
- プロジェクト状況を把握しやすくする

記載対象:

- 実装機能
- リファクタ
- 修正内容
- 破壊的変更

例:

```markdown
## 2026-05-17

### Added

- Google OAuth ログイン
- ユーザープロフィール画面

### Changed

- 認証ミドルウェアを整理

### Fixed

- セッション保持不具合を修正
```

# **Codex 運用ルール**

Codex は実装前に以下を必ず守る。

---

## **ルール1: 作業前にドキュメントを読む**

作業開始前に必ず以下を読む。

```
/docs/SPEC.md
/docs/TASKS.md
/docs/DECISIONS.md
/docs/CHANGELOG.md
```

ドキュメント未確認状態での実装は禁止。

---

## **ルール2: TASKS.md を作業の正とする**

Codex は:

1. 未完了タスクを選択する
2. 一度に1タスクだけ処理する
3. 完了してから次へ進む

複数の無関係タスクを同時並行しない。

---

## **ルール3: 完了時は TASKS.md を更新する**

完了したら:

```markdown
[ ] → [x]
```

に変更する。

例:

```markdown
- [x] Googleログインを実装する
```

更新せずに完了扱いすることは禁止。

---

## **ルール4: 設計判断は DECISIONS.md に残す**

重要な技術判断が発生した場合は:

```
/docs/DECISIONS.md
```

を更新する。

対象例:

- ライブラリ選定
- 認証方式
- キャッシュ戦略
- API設計方針
- 状態管理方式
- DBモデリング方針

軽微な実装詳細までは不要。

---

## **ルール5: 実装内容を CHANGELOG.md に記録する**

実装後は:

```
/docs/CHANGELOG.md
```

を更新する。

記録対象:

- 追加機能
- 挙動変更
- リファクタ
- 修正内容

---

## **ルール6: 勝手に大規模化しない**

禁止事項:

- 無断大規模リファクタ
- 不要なアーキテクチャ変更
- 過剰抽象化
- 理由なき新規フレームワーク導入

優先原則:

```
現在の要求を満たす最小構成
```

YAGNI を重視する。

---

## **ルール7: 既存ルールとの整合性を維持する**

新規コードは既存の:

- 命名規則
- ディレクトリ構成
- コーディングスタイル
- フレームワーク利用方針
- 依存方針

に合わせる。

矛盾する流儀を混在させない。

---

## **ルール8: 曖昧仕様への対応**

仕様が曖昧な場合は以下順で判断する。

```
1. 既存実装
2. DECISIONS.md
3. 最小安全実装
```

勝手な機能拡張は禁止。

---

## **ルール9: Backlog を管理する**

実装中に追加作業を発見した場合:

```markdown
## Backlog
```

へ追記する。

黙って放置しない。

---

## **ルール10: 重要知識をチャットだけに残さない**

重要情報はリポジトリへ保存する。

保存先:

- SPEC.md
- DECISIONS.md
- TASKS.md

禁止:

```
重要情報が会話ログにしか存在しない状態
```

---

# **推奨開発フロー**

```
1. ドキュメント確認
2. 次タスク選択
3. 実装
4. テスト/動作確認
5. TASKS.md 更新
6. CHANGELOG.md 更新
7. 必要なら DECISIONS.md 更新
8. コミット
```

---

# **タスク記述ルール**

推奨形式:

```markdown
- [ ] 機能Xを実装する
  - 完了条件:
    - 条件Aを満たす
    - 条件Bを満たす
    - テスト追加済み
```

曖昧タスクは禁止。

---

# **アーキテクチャ思想**

基本思想:

```
- 抽象化より単純性
- 理論的スケーラビリティより運用持続性
- 過剰設計より段階的成長
- 不要インフラ追加を避ける
- 巧妙さより保守性
- 低コスト優先
```

---

# **非目標**

SPEC.md に明記されない限り、以下は優先しない。

```
- 超大規模アクセス前提設計
- マイクロサービス化
- マルチリージョン
- 複雑イベント駆動構成
- エンタープライズ級抽象レイヤー
```

---

# **最終原則**

リポジトリ単体で以下が理解できる状態を維持する。

- 何が存在するか
- なぜ存在するか
- 何が未完了か
- なぜその設計になったか

会話履歴への依存を最小化すること。
